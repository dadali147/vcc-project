# 02.1-内部系统接口契约 v1

## 文档目标

本文件作为 VCC 内部系统接口契约的统一基线，用于对齐前端、后端、测试、产品对接口的共同理解。

## 适用范围

- 商户端接口
- 管理端接口
- 内部管理与风控接口
- 与 YeeVCC 上游交互后对内暴露的业务接口

## 契约原则

- 先定义业务语义，再定义字段。
- 先定义成功口径，再定义异常口径。
- 所有状态字段、枚举字段必须与状态字典统一。
- 所有分页、筛选、导出类接口使用统一结构。
- 敏感信息展示、三要素查看、OTP 等能力必须明确权限与审计要求。

## 统一请求约定

### 鉴权

- 登录后通过统一 token 机制鉴权。
- 商户主账户与子账户按角色权限控制接口可见范围。
- 管理端接口与商户端接口在权限域上严格隔离。

### 通用分页参数

- pageNum：页码，从 1 开始
- pageSize：每页条数
- orderByColumn：排序字段（可选）
- isAsc：排序方向 asc / desc（可选）

### 通用返回结构

```json
{
  "code": 200,
  "msg": "success",
  "data": {},
  "rows": [],
  "total": 0
}
```

## 模块级契约补充（第一批）

### 1. 认证与账户

#### 1.1 注册

**业务目标**：完成商户主账户注册，并创建后续登录、KYC、开卡流程所需的基础账户实体。

**入参建议**：merchantName、contactName、email、password、confirmPassword、country、inviteCode（可选）、emailCaptcha/code、agreeProtocol。

**出参建议**：userId、merchantId、registerStatus、nextStep、needKyc、token（如支持注册后自动登录）。

**状态建议**：REGISTERED / PENDING_REVIEW / REJECTED / ACTIVE。

**关键约束**：注册只允许通过邮箱验证码完成，不支持短信验证码注册；邮箱必须做唯一性校验；必须校验协议勾选；如存在邀请码或渠道码，需保留归因字段；注册成功后应明确是否直接进入 KYC。

**失败场景**：邮箱已存在、邮箱验证码错误、协议未勾选、渠道码无效、资料校验失败。

#### 1.2 登录

**业务目标**：完成商户端或管理端用户登录，返回统一会话凭证。

**入参建议**：username、password、captcha/code（如启用）、loginType（merchant/admin）。

**出参建议**：token、userId、userType、roles、permissions、nickName。

**失败场景**：用户不存在、密码错误、账户冻结、账户未完成必要认证、验证码错误。

**权限要求**：未登录可访问。

#### 1.3 获取当前用户信息

**业务目标**：给前端提供当前登录态下的身份、角色、菜单、权限。

**出参建议**：userInfo、roleGroup、permissions、profileStatus、kycStatus、registerStatus。

### 2. KYC 认证

#### 2.1 提交个人 KYC

**业务目标**：提交个人实名认证资料，并进入审核/自动校验流程。

**入参建议**：realName、idType、idNumber、nationality、idFrontImage、idBackImage、selfieImage、questionnaire。

**出参建议**：kycId、status（SUBMITTED / UNDER_REVIEW / REJECTED / APPROVED）、reviewType（AUTO / MANUAL）。

**关键约束**：图片资料必须可追溯；问卷字段与业务流程文档保持一致；审核状态必须与状态字典统一。

#### 2.2 提交企业 KYC

**入参建议**：companyName、companyNo、country、incorporationDoc、legalPersonName、legalPersonIdInfo、questionnaire。

**补充说明**：企业 KYC 默认进入人工审核链路，应保留企业主体与法人主体双重资料结构。

#### 2.3 查询 KYC 状态

**出参建议**：kycStatus、rejectReason、lastSubmitTime、reviewTime、reviewRemark。

### 3. 持卡人管理

#### 3.1 持卡人列表

**业务目标**：按商户维度查看全部持卡人，支持筛选和分页。

**筛选项建议**：holderName、status、phone、email、createTimeRange。

**列表字段建议**：holderId、holderName、mobile、email、status、relatedCardCount、createTime。

#### 3.2 新增持卡人

**关键字段建议**：firstName、lastName、mobile、email、country、address、city、state、postCode、idType、idNumber。

**约束**：同一商户下允许多个持卡人；不因姓名重复直接拦截；敏感身份信息按安全规则处理。

#### 3.3 持卡人详情

**出参建议**：基础信息、状态信息、名下卡片统计、最近操作记录。

### 4. 卡片管理

#### 4.1 卡片列表

**业务目标**：展示常规卡 / 共享卡列表，并支持状态、余额、限额维度筛选。

**筛选项建议**：cardType、cardStatus、holderId、cardBinId、keyword。

**列表字段建议**：cardId、cardName、cardType、holderName、cardNoMask、status、currency、balance、budgetAmount、availableAmount、createTime。

#### 4.2 卡片详情

**详情结构建议**：基本信息区、余额/预算区、敏感信息查看入口、Tabs（交易明细 / 操作记录）。

**返回字段建议**：cardId、cardType、cardStatus、cardNoMask、expiryMask、holderInfo、balanceInfo、feeInfo、limitInfo。

#### 4.3 开卡申请

**业务目标**：提交开卡任务，并返回任务状态。

**入参建议**：holderId、cardBinId、cardType（PREPAID / BUDGET）、cardName、budgetAmount（预算卡必填）、currency、spendScope。

**返回字段建议**：taskId、applyStatus、estimatedFinishTime。

**关键约束**：预算卡必须校验预算额度；储值卡与预算卡资金路径必须遵循业务规则；开卡费用承担方必须明确。

#### 4.4 敏感信息查看

**业务目标**：在合规和权限控制下查看卡号 / CVV / 有效期。

**出参建议**：cardNumber、cvv、expiryDate、aesVersion、viewExpireAt。

**审计要求**：必须记录查看人、查看时间，并设置短时有效期。

#### 4.5 OTP 读取

**业务目标**：展示上游回调下发的验证码，供商户完成支付验证。

**出参建议**：otpCode、expireTime、sourceType、receivedTime。

**关键约束**：OTP 展示必须受权限控制；过期验证码不可继续展示为可用状态；OTP 来源必须能追溯到 webhook 记录。

## 模块级契约补充（第二批）

### 5. 充值与余额

#### 5.1 充值申请

**业务目标**：对卡片发起充值，并返回同步或异步处理结果。

**入参建议**：cardId、amount、currency、orderNo、remark、sourceAccountId。

**出参建议**：orderNo、upstreamOrderNo、status（SUCCESS / PENDING / FAILED）、balance、finishTime。

**关键约束**：必须校验账户余额；必须区分同步充值与异步充值；必须保留商户订单号和上游订单号映射关系。

#### 5.2 充值记录

**筛选项建议**：cardId、orderNo、status、timeRange。

**列表字段建议**：orderNo、cardId、amount、currency、status、sourceAccountName、finishTime、createTime。

#### 5.3 账户余额查询

**出参建议**：accountId、availableBalance、frozenBalance、currency、lastUpdateTime。

**关键约束**：账户余额应与卡充值、预算扣减逻辑保持一致。

#### 5.4 卡片余额查询

**出参建议**：cardId、balance、availableAmount、currency、lastTxnTime。

**补充说明**：储值卡与预算卡的余额语义不同，前端展示口径必须区分。

### 6. 交易与账单

#### 6.1 交易记录列表

**业务目标**：查询交易流水，支持卡片、时间、状态、类型维度筛选。

**筛选项建议**：cardId、txnType、status、merchant、timeRange。

**列表字段建议**：txnId、cardId、cardNoMask、txnType、amount、currency、merchantName、status、txnTime。

#### 6.2 交易详情

**出参建议**：txnId、authAmount、settleAmount、feeAmount、merchantInfo、country、mcc、statusFlow、rawType。

**关键约束**：需明确授权、清算、退款等不同交易阶段的映射关系。

#### 6.3 对账单导出

**入参建议**：statementMonth、cardId、fileLang、exportType。

**出参建议**：taskId、fileId、downloadUrl、expireTime。

**补充说明**：导出类接口建议统一走异步任务模式。

#### 6.4 下载中心记录

**列表字段建议**：fileId、fileName、fileType、language、status、createTime、expireTime。

### 7. 预算与限额

#### 7.1 共享卡预算设置

**业务目标**：设置或调整共享卡预算额度。

**入参建议**：cardId、budgetAmount、effectiveType、effectiveTime、remark。

**出参建议**：cardId、budgetAmount、availableBudget、effectiveTime、status。

**关键约束**：预算调整必须产生审计记录；调整后额度不得突破账户真实承载能力。

#### 7.2 限额调整记录

**列表字段建议**：recordId、cardId、beforeAmount、afterAmount、operatorName、operateTime、remark。

#### 7.3 限额历史查询

**出参建议**：historyList，字段包括调整前额度、调整后额度、操作时间、操作人、备注。

### 8. 管理后台

#### 8.1 用户管理

**业务目标**：查看商户、子账户及状态信息。

**列表字段建议**：userId、merchantName、accountType、status、kycStatus、createTime。

#### 8.2 KYC 审核

**出参建议**：kycId、subjectType、submitterName、reviewStatus、reviewerName、submitTime、reviewTime。

**审核动作建议**：approve、reject、requestMoreInfo。

#### 8.3 卡 BIN 管理

**列表字段建议**：binId、binCode、productCode、currency、scene、status。

**关键约束**：BIN 配置变更必须影响开卡可选项和费率规则。

#### 8.4 费率配置

**字段建议**：feeType、cardType、currency、chargeMode、feeValue、effectiveStatus。

**关键约束**：必须支持区分储值卡/预算卡的不同扣费路径。

#### 8.5 风控配置

**字段建议**：ruleId、ruleName、ruleType、threshold、status、effectiveTime。

**补充说明**：需支持 MCC、地区、频次、金额等维度规则。

#### 8.6 系统日志

**列表字段建议**：logId、operatorName、moduleName、actionType、resultStatus、requestTime。

**关键约束**：敏感信息查看、预算调整、审核动作必须可审计。

## 权限矩阵（简版）

### 商户主账户

- 可查看本商户全部持卡人、卡片、交易、充值记录
- 可发起开卡、充值、预算调整
- 可查看敏感信息和 OTP（需满足附加权限或二次校验）
- 可管理子账户

### 商户子账户

- 默认仅能访问授权范围内的数据
- 是否可开卡、充值、查看敏感信息，由角色权限单独控制
- 默认不可进行费率配置、风控配置、BIN 管理等后台动作

### 管理端账户

- 可查看全局商户、KYC、卡 BIN、费率、风控、日志信息
- 不直接承担商户端业务操作语义
- 高风险动作必须保留审核和审计链路

## 模块级契约补充（第三批）

### 1.4 子账户管理

**业务目标**：支持商户主账户对本商户下子账户进行创建、授权、禁用、重置密码与数据范围控制，保证"谁能看什么、谁能操作什么"可被清晰约束与审计。

#### 1.4.1 子账户列表

**业务目标**：分页查看本商户全部子账户及其状态、角色、数据权限范围。

**筛选项建议**：keyword、status、roleCode、dataScopeType、createTimeRange。

**列表字段建议**：subAccountId、loginEmail、userName、status、roleCode、roleName、dataScopeType、lastLoginTime、createTime。

**关键约束**：

- 商户主账户仅可查看本商户子账户，不可跨商户查询。
- 已删除子账户默认不展示，除非显式传入 includeDisabled。
- 列表返回的数据范围必须与当前登录人的管理权限一致。

#### 1.4.2 新增子账户

**入参建议**：userName、loginEmail、mobile（可选）、roleIds、dataScopeType、dataScopeValue、remark、initPasswordMode。

**出参建议**：subAccountId、status、tempPasswordExpired、needResetPassword。

**字段约束**：

- loginEmail 必填且商户内唯一，作为登录账号。
- roleIds 至少选择一个；角色为空时不允许创建。
- dataScopeType 建议枚举：ALL / HOLDER / CARD / DEPARTMENT / CUSTOM。
- dataScopeValue 在 dataScopeType 为 CUSTOM / HOLDER / CARD / DEPARTMENT 时必填。
- initPasswordMode 建议枚举：SYSTEM_RANDOM / MANUAL_SET。

**失败场景**：邮箱已存在、角色不存在、数据范围越权、主账户无权限创建、初始密码不符合安全规则。

**审计要求**：必须记录创建人、被创建账号、角色集合、数据范围、创建结果。

#### 1.4.3 子账户详情

**出参建议**：subAccountId、userName、loginEmail、mobile、status、roleList、dataScopeType、dataScopeValue、lastLoginTime、createTime、updateTime。

**补充说明**：详情接口必须返回"角色权限摘要 + 数据权限摘要"，避免前端二次拼装口径不一致。

#### 1.4.4 编辑子账户

**入参建议**：subAccountId、userName、mobile、roleIds、dataScopeType、dataScopeValue、remark。

**关键约束**：

- 不允许通过编辑接口直接更换归属 merchantId。
- 已禁用账户允许修改基础资料，但不允许在未恢复前执行高风险授权。
- 若角色与数据范围同时变化，应在一次事务内完成，避免出现中间态越权。

**出参建议**：subAccountId、status、updatedFields。

#### 1.4.5 启用 / 禁用子账户

**业务目标**：切换子账户登录和操作状态。

**入参建议**：subAccountId、targetStatus、reason。

**targetStatus 建议枚举**：ACTIVE / DISABLED / LOCKED。

**关键约束**：

- 禁用后应立即阻断登录态续期；已签发 token 应在可控时效内失效。
- 默认不允许子账户禁用商户主账户。
- LOCKED 更适用于密码连续失败或风控冻结；DISABLED 更适用于人工停用。

**出参建议**：subAccountId、beforeStatus、afterStatus、effectiveTime。

#### 1.4.6 重置子账户密码

**业务目标**：由商户主账户或具备授权的管理员重置子账户密码。

**入参建议**：subAccountId、resetMode、newPassword（手动模式时必填）、notifyByEmail。

**resetMode 建议枚举**：SYSTEM_RANDOM / MANUAL_SET / FORCE_RESET_NEXT_LOGIN。

**出参建议**：subAccountId、resetResult、needResetPassword、tempPasswordExpired。

**安全约束**：

- 不回显明文密码；如为随机密码，仅允许一次性安全通道下发。
- 重置后必须清理旧登录态。
- 必须记录操作人、目标账号、重置模式、结果状态。

#### 1.4.7 子账户权限查询

**业务目标**：为前端权限树、菜单控制、按钮显隐提供统一权限快照。

**出参建议**：subAccountId、roleList、permissionKeys、menuKeys、dataScopeType、dataScopeValue。

**补充说明**：permissionKeys 应直接对齐前端按钮级权限；menuKeys 应对齐路由与页面显隐。

#### 1.4.8 子账户接口权限矩阵补充

- 商户主账户：可创建、编辑、禁用、重置密码、分配角色、配置数据范围。
- 有授权的高级子账户：可查看子账户列表与详情，但是否可编辑/禁用需单独授权。
- 普通子账户：默认不可访问子账户管理相关接口。

### 6.5 操作记录接口

**业务目标**：沉淀卡片、持卡人、预算、账户等对象的关键人工操作轨迹，供详情页追溯、风控排查、客服核单与审计使用。

#### 6.5.1 操作记录列表

**适用对象**：cardId、holderId、subAccountId、merchantId、taskId。

**入参建议**：targetType、targetId、actionType（可选）、operatorType（可选）、pageNum、pageSize、startTime、endTime。

**targetType 建议枚举**：CARD / HOLDER / SUB_ACCOUNT / MERCHANT / TASK / KYC。

**actionType 建议枚举**：CREATE / UPDATE / FREEZE / UNFREEZE / CLOSE / RECHARGE / SET_BUDGET / VIEW_SENSITIVE / VIEW_OTP / RESET_PASSWORD / APPROVE / REJECT / EXPORT。

**列表字段建议**：recordId、targetType、targetId、actionType、actionName、operatorId、operatorName、operatorType、resultStatus、requestTime、clientIp、traceId。

**关键约束**：

- 操作记录按时间倒序返回。
- 同一 traceId 下的多步动作应支持聚合查看。
- 记录展示层不应暴露完整敏感值，只展示脱敏摘要与字段变化说明。

#### 6.5.2 操作记录详情

**出参建议**：recordId、targetType、targetId、actionType、operatorInfo、beforeSnapshot、afterSnapshot、resultStatus、failureReason、requestPayloadMask、responsePayloadMask、requestTime、finishTime、traceId。

**使用场景**：卡片详情页"操作记录"Tab、KYC 审核追溯、预算调整核查、敏感信息访问审计。

**安全约束**：

- beforeSnapshot / afterSnapshot 中涉及卡号、CVV、证件号、邮箱、手机号的字段必须脱敏。
- requestPayloadMask / responsePayloadMask 只保留排查必要字段，不回传完整上游密文。

#### 6.5.3 卡片详情页操作记录口径

- 默认展示与当前 cardId 直接相关的人工操作。
- 必含动作：开卡申请、冻结、解冻、销卡、充值、预算调整、敏感信息查看、OTP 查看。
- 如动作由异步任务触发，应同时返回关联 taskId 与 taskStatus，方便前端跳转任务详情。

#### 6.5.4 审计落库建议

**建议字段**：recordId、targetType、targetId、actionType、operatorId、operatorType、beforeValueMask、afterValueMask、resultStatus、errorCode、errorMessageMask、requestTime、finishTime、clientIp、traceId、sourceSystem。

**与"系统日志"关系**：系统日志偏全局后台行为；操作记录偏业务对象维度追溯。两者可共源但不可混成一个前端展示口径。

## 统一异步任务查询结构

### 业务目标

统一承接注册审核、开卡申请、充值处理、对账单导出等存在异步完成语义的接口，避免每个页面单独定义轮询结构。

### 统一适用范围

- 注册结果查询（如需人工审核或邮件激活）
- 开卡结果查询
- 充值结果查询
- 对账单导出任务查询
- 其他需"提交后稍后完成"的内部任务

### 统一任务状态枚举

| 状态 | 说明 |
|------|------|
| INIT | 任务已创建，尚未下发执行 |
| PROCESSING | 处理中 |
| SUCCESS | 处理成功 |
| FAILED | 处理失败 |
| PARTIAL_SUCCESS | 部分成功（适用于批量任务） |
| CANCELED | 任务已取消 |
| EXPIRED | 任务结果已过期或下载地址已过期 |

### 统一查询接口建议

**入参建议**：taskId、taskType。

**taskType 建议枚举**：REGISTER_REVIEW / CARD_APPLY / RECHARGE / STATEMENT_EXPORT / BATCH_UPDATE。

**出参建议**：taskId、taskType、taskStatus、businessStatus、percent、message、errorCode、errorMessage、resultData、createdTime、updatedTime、finishedTime、expireTime。

**字段说明**：

- taskStatus：任务执行态，使用统一枚举。
- businessStatus：业务语义态，例如开卡中的 APPLY_SUCCESS / APPLY_REJECTED。
- percent：进度百分比，无法计算时可返回 null。
- resultData：不同业务的结果载荷，如 cardId、orderNo、downloadUrl。
- expireTime：适用于导出文件、临时查看结果等具备有效期的任务。

### 统一返回示例

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "taskId": "TASK202603220001",
    "taskType": "CARD_APPLY",
    "taskStatus": "PROCESSING",
    "businessStatus": "UPSTREAM_SUBMITTED",
    "percent": 60,
    "message": "上游已受理，等待制卡结果",
    "errorCode": null,
    "errorMessage": null,
    "resultData": null,
    "createdTime": "2026-03-22T01:10:00+08:00",
    "updatedTime": "2026-03-22T01:12:00+08:00",
    "finishedTime": null,
    "expireTime": null
  }
}
```

### 业务模块映射建议

- **开卡申请**：提交接口返回 taskId，查询接口返回 cardId / applyStatus / rejectReason。
- **充值申请**：提交接口在 PENDING 时返回 taskId 或 orderNo，查询接口返回充值最终状态与余额结果。
- **对账单导出**：提交接口返回 taskId，查询接口返回 fileId、downloadUrl、expireTime。
- **注册审核**：如注册需进入人工审核，则查询接口返回 registerStatus、rejectReason、nextStep。

### 前端交互约束

- 页面不得硬编码各自不同的轮询状态枚举，统一使用 taskStatus + businessStatus 双层结构。
- 当 taskStatus = SUCCESS 但业务失败时，应通过 businessStatus / errorCode 呈现业务拒绝原因，避免把"任务执行成功"误解为"业务成功"。
- 导出类结果若 expireTime 已过，应提示重新发起任务，而不是继续使用旧 downloadUrl。

## 模块级契约补充（第四批）

### 1.1.1 发送邮箱验证码

**业务目标**：为注册、邮箱绑定、密码找回等场景发送邮箱验证码，作为邮箱归属验证前置步骤。

**入参建议**：email、bizType、captcha（如启用图形验证码）、traceId（可选）。

**bizType 建议枚举**：REGISTER / RESET_PASSWORD / BIND_EMAIL / SECURITY_VERIFY。

**出参建议**：requestId、expireTime、retryAfter、channel。

**关键约束**：

- 注册场景默认使用邮箱验证码，不支持短信验证码替代注册主链路。
- 同一邮箱同场景需做发送频控；应返回 retryAfter 供前端倒计时使用。
- channel 建议固定返回 EMAIL，后续如有多渠道再扩展。

**失败场景**：邮箱格式非法、发送过于频繁、图形验证码错误、邮箱已注册（注册场景）、邮件服务异常。

### 3.4 BIN 下拉来源接口

**业务目标**：为开卡页提供可选 BIN / 发卡产品下拉列表，并附带卡类型、币种、费用、适用场景等前端必要信息。

**筛选入参建议**：cardType（可选）、currency（可选）、scene（可选）、status（默认 ENABLED）。

**列表字段建议**：cardBinId、binCode、productCode、productName、cardType、currency、scene、issuerName、openCardFee、monthlyFee、chargeMode、status。

**关键约束**：

- 返回结果必须只包含当前商户有权限申请的 BIN，不允许把停用或不适用产品混进下拉。
- 开卡页展示项可用 productName + cardType + currency 组合，但真实值提交统一使用 cardBinId。
- 如 BIN 受商户等级、KYC 状态、国家限制影响，接口必须在返回层直接过滤，而不是让前端猜。

### 4.1.1 首页聚合接口

**业务目标**：为商户端首页统一提供账户总览、KYC 进度、卡片与持卡人核心指标、待办提醒和快捷入口摘要。

**入参建议**：none。

**出参建议**：merchantId、merchantName、registerStatus、kycStatus、availableBalance、frozenBalance、cardCount、activeCardCount、holderCount、pendingTaskCount、unreadMessageCount、lastLoginTime、todoList。

**todoList 子项建议**：todoType、title、summary、targetType、targetId、priority。

**关键约束**：

- 首页聚合接口允许聚合返回，但指标字段名称必须可追溯到真实业务字段，不允许只返回"overviewData"黑盒。
- pendingTaskCount 需明确是否包含开卡、充值、导出等任务；建议至少覆盖对商户可感知的异步任务。
- 首页待办应直接绑定 registerStatus / kycStatus / taskStatus / messageReadStatus，不允许前端自己写业务判断树。

### 4.3.1 开卡记录查询接口

**业务目标**：分页查询开卡申请历史、任务执行状态与最终业务结果，为开卡记录页和任务结果页提供统一数据源。

**筛选入参建议**：taskId、holderId、cardType、applyStatus、taskStatus、startTime、endTime。

**列表字段建议**：taskId、holderId、holderName、cardType、cardBinId、applyStatus、taskStatus、cardId、rejectReason、errorCode、createdTime、finishedTime。

**详情补充字段建议**：estimatedFinishTime、upstreamApplyId、traceId、resultMessage。

**关键约束**：

- 必须同时返回 applyStatus 与 taskStatus，禁止继续只给一个 status。
- 当 taskStatus = SUCCESS 但业务失败时，应通过 applyStatus = REJECTED / FAILED + rejectReason / errorCode 呈现真实结果。
- cardId 在开卡成功时应尽量直接返回，便于页面跳详情。

### 6.4.1 消息通知接口

**业务目标**：为消息通知页提供列表、详情、已读处理、未读数统计与业务跳转所需信息。

#### 6.4.1.1 消息列表

**筛选入参建议**：messageType、messageReadStatus、startTime、endTime。

**列表字段建议**：messageId、messageType、title、summary、messageReadStatus、targetType、targetId、taskId、sendTime、readTime、channels。

**关键约束**：

- messageReadStatus 统一使用正式枚举，不再裸用 readStatus/status。
- 消息跳转依赖 targetType + targetId + taskId，至少返回其中能闭环的一组。
- channels 建议返回 notificationChannel 数组，反映真实投递渠道。

#### 6.4.1.2 消息详情

**出参建议**：messageId、messageType、title、content、messageReadStatus、targetType、targetId、taskId、securityEventType、sendTime、readTime、channels。

**补充说明**：高风险消息详情建议返回 securityEventType，便于跳转安全日志或风险详情。

#### 6.4.1.3 标记已读

**入参建议**：messageId 或 messageIds。

**出参建议**：successCount、failCount、readTime。

#### 6.4.1.4 未读数查询

**出参建议**：totalUnreadCount、groupByType（messageType 维度统计，可选）。

### 1.3.1 个人资料更新接口

**业务目标**：支持当前登录用户更新基础个人资料，不与密码修改、安全操作混在一个提交模型中。

**入参建议**：nickName、userName、mobile、avatarUrl（可选）。

**出参建议**：userId、nickName、userName、mobile、updateTime。

**关键约束**：

- email 如作为登录主账号，默认不通过本接口直接修改；如后续允许改邮箱，必须走验证码二次校验链路。
- 资料更新接口不得承接密码修改、角色调整、数据权限变更等安全级动作。

**失败场景**：手机号格式错误、昵称非法、资料重复、无登录态。

### 8.6.1 安全日志接口

**业务目标**：为个人设置 / 安全设置页提供登录记录、安全事件记录、异常行为回溯。

**筛选入参建议**：securityEventType、startTime、endTime、resultStatus。

**列表字段建议**：logId、securityEventType、eventName、resultStatus、clientIp、deviceInfo、location、requestTime、traceId。

**详情字段建议**：logId、securityEventType、resultStatus、clientIp、deviceInfo、location、requestTime、finishTime、traceId、failureReason、operatorId、targetId。

**关键约束**：

- 高风险事件必须覆盖 PASSWORD_CHANGED、PASSWORD_RESET、SENSITIVE_VIEW、OTP_VIEW、ACCOUNT_LOCKED、DEVICE_CHANGED、RISK_TRIGGERED。
- 安全日志与消息通知可联动，但接口语义不能混成一个列表。
- clientIp、deviceInfo、location 是否全部向商户端展示，可按安全策略脱敏，但日志内必须保留。

- 发送邮箱验证码接口必填：email、bizType；条件必填：captcha（当图形验证码开启时）。
- 开卡记录查询接口选填筛选：taskId、holderId、cardType、applyStatus、taskStatus、startTime、endTime。
