# 02.3-状态与字段统一收口记录 v1

## 文档目标

把《01.3-业务规则-状态字典 v1》《01.4-页面基线-页面清单与字段》《02.1-内部系统接口契约 v1》中已经出现但仍散落、冲突、未定稿的状态与字段问题统一压成一份收口清单，作为下一轮正式定稿和派单执行的唯一入口。

## 收口范围

- 状态族统一
- 页面字段命名统一
- 聚合对象展开清单
- 双状态/双语义字段边界统一
- 页面跳转与按钮显隐依赖统一

## 一、已完成的统一收口（本轮）

### 状态族已补入《01.3》第一版

- 已补：`holderStatus`、`rechargeStatus`、`transactionStatus`、`downloadStatus`、`subAccountStatus`、`auditActionType`。
- 现状：01.3 已覆盖注册、KYC、卡片、开卡申请、异步任务、预算、持卡人、充值、交易、下载、子账户、审计动作。
- 意义：状态字典不再只停留在前 6 组核心状态，页面和接口可以开始按正式名字对齐。

### 页面清单已补入《01.4》第一版

- 已补页面：仪表盘首页、卡片列表、开卡记录、记录中心、下载中心、子账户管理、消息通知、个人设置 / 安全设置。
- 现状：01.4 已覆盖注册、登录、KYC、用卡人管理、卡片详情、开卡、充值，以及上述新增 8 页。
- 意义：页面基线不再只有页面名，已经具备可对齐的字段骨架。

### 双层状态语义已明确

- 已明确的双层结构：
  - 开卡：`taskStatus`（任务执行态） + `applyStatus`（开卡业务态）
  - 充值：`taskStatus`（任务执行态） + `rechargeStatus`（充值业务态）
  - 下载：`taskStatus`（任务执行态） + `downloadStatus`（文件可用态）
- 结论：以后不能再用一个 `status` 糊过去。

## 二、统一收口清单（按优先级）

### P1-01：status 泛字段全面禁用，改为对象级状态字段

**问题：** 01.4 和 02.1 中仍存在 `status` 的宽泛写法，容易导致前端/后端/测试各自脑补。

**已明确替代：**

| 对象 | 推荐字段名 |
|------|-----------|
| 持卡人 | `holderStatus` |
| 充值 | `rechargeStatus` |
| 子账户 | `subAccountStatus` |
| 交易 | `transactionStatus` |
| 卡片 | `cardStatus` |
| 下载文件 | `downloadStatus` |
| 任务 | `taskStatus` |

**要求：** 后续接口、页面、测试用例、数据库字段说明禁止继续直接写裸 `status`，必须写对象级状态名。

### P1-02：taskStatus 不等于业务成功，必须绑定 businessStatus / applyStatus / rechargeStatus / downloadStatus

**问题：** 02.1 已明确 `taskStatus`，但页面层仍容易把 `SUCCESS` 误读为业务成功。

**收口结论：**
- `taskStatus` = `SUCCESS` 只代表任务执行完成。
- 业务成功与否，必须再看 `businessStatus` 或对应业务状态字段。
- 影响页面：开卡记录、充值记录、下载中心、记录中心、首页待办。

### P1-03：页面状态显隐一律绑定正式状态字典，不允许组件内写死

**问题：** 01.4 已多次强调，但还没有统一落成硬规则。

**收口结论：**
- 卡片操作按钮绑定 `cardStatus`。
- 开卡结果展示绑定 `applyStatus` + `taskStatus`。
- 子账户登录/编辑/禁用按钮绑定 `subAccountStatus`。
- 消息跳转与待办展示绑定 `registerStatus` / `kycStatus` / `taskStatus` / `readStatus`。

### P1-04：展示字段与真实字段必须拆开

**问题：** 页面层存在 `holderName`、`summary`、指标聚合值等展示口径，如果不拆，会反向污染接口。

**收口结论：**
- `holderName` 是展示字段，真实入参仍为 `firstName` + `lastName`。
- 首页指标可以聚合展示，但接口应保留真实基础字段来源。
- 记录中心的 `recordId` 是页面统一字段，不等于底层真实主键名统一。
- **原则：** 展示字段可存在，但必须注明真实字段映射。

### P1-05：消息通知与消息已读态尚未进入正式状态字典

**问题：** 01.4 已出现 `messageType`、`readStatus`，但 01.3 尚未正式收口。

**收口结论：** 下一轮必须补入：
- `messageType`（如 `SYSTEM` / `TASK` / `KYC` / `BILLING` / `SECURITY`）
- `messageReadStatus`（`UNREAD` / `READ`）
- 否则消息页面只能继续靠前端自由发挥。

### P2-01：聚合对象必须继续展开，不允许长期停留在 object 黑盒

**当前未展开对象：**
- `holderInfo`
- `balanceInfo`
- `feeInfo`
- `limitInfo`
- `questionnaire`
- `legalPersonIdInfo`
- 首页统计聚合对象
- 权限树对象（`roleList` / `permissionKeys` / `menuKeys` / `dataScope` 摘要）

**收口结论：**
- 01.4 下一轮必须把这些对象至少拆到二级字段。
- 02.1 下一轮必须明确这些对象的字段类型、必填性、脱敏规则。

### P2-02：页面与接口缺独立成文的模块仍需补章节

**当前缺正文接口：**
- 首页聚合接口
- 开卡记录查询接口
- 消息通知相关接口
- 个人资料更新接口
- 登录设备/安全日志接口
- BIN 下拉来源接口
- 发送邮箱验证码接口

**结论：** 这些页面虽已有字段骨架，但没有对应 02.1 正文，仍不算真正收口。

### P2-03：跳转规则和 nextStep 仍未正式枚举化

**问题：** 注册、KYC、任务结果、消息通知都涉及跳转，但目前只散落在描述句。

**待收口字段：**
- `nextStep`
- `targetType`
- `targetId`
- 可能还需要 `targetPage` / `actionCode`

**结论：** 下一轮应把"状态 -> 页面跳转 -> 按钮动作"整理成矩阵，否则前端还会继续写 if-else。

### P2-04：交易阶段与上游语义映射仍未钉死

**问题：** `transactionStatus` 已补第一版，但与上游 `rawType` / 授权/清算/退款链路还没完全对齐。

**结论：** 需要把 `transactionStatus`、`statusFlow`、`rawType` 三者关系明确下来，避免交易页、账单页、客服页各说各话。

### P2-05：downloadStatus 与 taskStatus 的字段归属还需数据库/接口双向钉死

**问题：** 下载中心页面已经需要两套状态，但 02.1 和数据库设计还未明确由谁主承载。

**结论：**
- `taskStatus` 用于异步执行表。
- `downloadStatus` 用于文件结果表或文件元数据字段。
- 如两者共表，也必须在字段语义上拆清楚。

## 三、字段命名统一结论（第一版）

### 统一保留

| 字段名 | 用途 |
|--------|------|
| `emailCaptcha` | 注册邮箱验证码 |
| `captcha` | 登录图形验证码 |
| `username` | 登录账号统一字段 |
| `holderStatus` | 持卡人状态 |
| `rechargeStatus` | 充值状态 |
| `subAccountStatus` | 子账户状态 |
| `transactionStatus` | 交易状态 |
| `downloadStatus` | 下载文件状态 |
| `taskStatus` | 任务执行状态 |
| `applyStatus` | 开卡申请业务状态 |
| `registerStatus` | 注册状态 |
| `kycStatus` | KYC 状态 |
| `cardStatus` | 卡片状态 |
| `budgetStatus` | 预算状态 |

### 明确不再推荐继续扩散的写法

- `status`（裸字段）
- `code` / `captchaCode`（用于注册验证码）
- 多套"账号/邮箱/用户名"并行字段
- 把 `SUCCESS` / `FAILED` 直接当交易、开卡、充值、下载统一业务状态

## 四、对下游执行的硬约束

1. 任何新页面、新接口、新表字段，如果涉及状态或枚举：先补 01.3，再进 02.1 / 01.4。
2. 任何页面新增展示字段：必须注明真实字段名，不允许只有 UI 文案。
3. 任何异步业务：必须同时说明 `taskStatus` 和业务状态字段，不允许只返回一个 `status`。
4. 任何高风险动作：必须同步考虑 `auditActionType`、`operatorId`、`targetId`、`traceId`、`resultStatus`。

## 五、下一轮剩余缺口

1. 01.3 仍缺：`messageType`、`messageReadStatus`、可能的 `notificationChannel` / `securityEventType`。
2. 01.4 仍缺：聚合对象字段展开、按钮权限矩阵、页面跳转矩阵、空状态/异常状态文案。
3. 02.1 仍缺：首页聚合、开卡记录查询、消息通知、个人资料更新、安全日志、验证码发送、BIN 来源接口等独立章节。
4. 数据库设计仍未完成逐表对齐，尤其是 `downloadStatus`、`transactionStatus`、`subAccountStatus`、`holderStatus` 的真实落库位置。
5. 错误码与状态映射仍未做完整矩阵，当前只有散点示例。

## 当前结论

- 本轮已经把"状态叫什么、页面字段叫什么、哪些地方不能再用裸 status"这件事往前推了一大步。
- 但别自我感动，现在只是把散点压成了可执行清单，还没到最终定稿。下一轮该干的很明确：继续展开对象、补独立接口、补跳转矩阵、补数据库对齐。

## 六、本轮新增收口（B-继续推进）

### P1-06：消息与安全类枚举已正式补入 01.3

- 已补：`messageType`、`messageReadStatus`、`notificationChannel`、`securityEventType`。
- 影响范围：消息通知页、安全设置页、首页待办、任务结果通知、KYC 结果通知、审计与安全日志。
- 收口结论：
  - 消息分类、已读状态、通知通道、安全事件类型不再允许页面和接口各写各的自由文案。
  - 高风险消息应优先走 `SECURITY_ALERT` + `securityEventType`，站内信/邮件等通道用 `notificationChannel` 表达，不要混字段语义。

### P1-07：01.4 六个聚合对象已展开到字段级

- 已补：`holderInfo`、`balanceInfo`、`feeInfo`、`limitInfo`、`questionnaire`、`legalPersonIdInfo`。
- 收口结论：
  - 这些对象不再允许长期作为黑盒 object 出现在页面和接口契约里。
  - 页面层已经具备字段级对齐基础；下一步应在 02.1 / 数据库设计中继续钉类型、必填、脱敏规则。

### P1-08：02.1 已补 7 个缺口接口章节
