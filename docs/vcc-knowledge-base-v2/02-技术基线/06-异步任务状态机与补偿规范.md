# 02.6-异步任务状态机与补偿规范

- **文档版本：** v1.0
- **更新时间：** 2026-03-22
- **状态：** 首版正式正文
- **定位：** 02 区技术基线正式规范，承接 02.1 接口契约中的异步任务处理要求，并约束状态流转、轮询/回调协同、幂等、补偿、超时和人工介入。

## 一、文档目标与适用范围

### 1.1 文档目标

本文档用于统一 VCC 内部系统对异步任务的设计、落库、推进、补偿与验收口径，解决以下问题：

- 上游接口存在同步返回、异步完成、回调补充通知等混合模式，若无统一状态机，业务状态容易漂移。
- 开卡、充值、转出等关键资金与卡生命周期动作存在"已受理但未最终完成"的窗口，必须有标准任务模型承接。
- 在轮询查询、Webhook 回调、人工重试并存的情况下，必须明确谁负责推进、谁负责兜底、谁负责终态判定。
- 失败不能只做"报错返回"，必须区分可重试失败、待补偿失败、人工介入失败和最终关闭失败。

### 1.2 适用范围

本文档适用于以下场景：

- 基于 YeeVCC 上游接口发起、且存在异步确认可能性的业务动作
- 内部系统需要记录任务主表、状态流转、重试记录、补偿记录、审计日志的场景
- 同时接受查询轮询与 Webhook 回调推进的任务场景

**当前首版强制纳入的典型任务：**
- 开卡任务
- 充值任务
- 转出任务

**后续可复用到以下场景：**
- 销卡、冻结/解冻
- 敏感信息申请与失效重取
- 三方授权确认
- 人工审核后继续推进的风控任务

### 1.3 与其他文档的边界

- **02.1《内部系统接口契约 v1》**：定义接口、字段、上游任务返回方式与查询路径；不负责统一内部状态机落库口径。
- **02.5《Webhook 与异步任务规范承接方案》**：定义事件接入、回调验签、事件处理链路；不替代任务状态机本体。
- **03《数据库设计文档》**：定义持久化表结构基线；本规范补充异步任务主表、状态变更日志、补偿记录等设计要求。

## 二、异步任务总览

### 2.1 异步任务定义

异步任务，是指"请求已被内部系统或上游系统接收，但业务结果尚未在当前请求响应内最终确认"的任务单元。

异步任务必须具备以下特征：
- 有唯一任务标识，可追踪整条链路
- 有明确发起时间、推进时间、终态时间
- 有可枚举的状态集合与合法流转规则
- 有结果来源：同步结果、轮询查询、Webhook 回调、人工处理中的至少一种
- 有失败分类与补偿动作定义

### 2.2 统一设计原则

- **业务动作与异步任务分离：** 业务单据是业务视角，异步任务是执行视角，二者不得混为一个 `status` 字段。
- **先落任务，再调上游：** 凡存在异步可能的动作，内部先创建任务记录，再发起上游请求，避免请求出去了但内部无账可查。
- **终态唯一：** 任一任务只能收敛到一个最终终态，不允许既成功又失败。
- **推进多源，判定单源：** 轮询、回调、人工都可以推动任务，但最终状态必须由统一规则判断并落库。
- **先幂等，再重试：** 所有异步推进逻辑必须先去重校验，再执行状态推进，禁止靠"重复调用碰运气"。
- **资金类优先审计：** 充值、转出、余额修正类任务必须保留完整审计证据。

### 2.3 任务参与角色

- **任务发起方：** 内部业务服务，负责创建任务并发起上游调用
- **状态推进器：** 轮询任务、Webhook 处理器、人工干预动作
- **状态判定器：** 统一状态机规则，负责校验是否允许迁移
- **补偿执行器：** 负责重试、回滚、对账、人工挂起、人工关闭
- **审计查看方：** 运营、财务、研发、QA

## 三、统一状态机定义

### 3.1 标准状态集合

首版统一状态机如下：

| 状态 | 含义 | 是否终态 |
|------|------|---------|
| `INIT` | 任务已创建，尚未发起上游请求 | 否 |
| `SUBMITTING` | 正在提交上游请求，尚未确认受理结果 | 否 |
| `ACCEPTED` | 上游已受理，等待最终结果 | 否 |
| `PROCESSING` | 已进入处理中，等待轮询或回调推进 | 否 |
| `SUCCESS` | 任务执行成功，业务结果已确认 | 是 |
| `FAILED_RETRYABLE` | 本次执行失败，但允许系统自动重试或补偿重试 | 否 |
| `FAILED_WAIT_COMPENSATE` | 已确认异常，等待补偿动作或对账确认 | 否 |
| `FAILED_MANUAL` | 系统无法自动收敛，需要人工介入 | 否 |
| `CANCELLED` | 任务被业务主动取消或关闭 | 是 |
| `TIMEOUT` | 超过约定时限仍未得到最终结果，进入超时终态/超时分支 | 视任务类型而定 |
| `CLOSED` | 任务已关闭，且后续不得再推进 | 是 |

> 说明：
> - 对资金敏感任务，不建议直接把 `TIMEOUT` 当作最终业务失败；通常应先进入 `FAILED_WAIT_COMPENSATE` 或 `FAILED_MANUAL` 再由补偿闭环收口。
> - `CLOSED` 用于表达流程结束且不再允许进一步补偿推进的关闭态；并非所有任务都必须用到。

### 3.2 推荐主流转路径

**标准主路径：**

```
INIT -> SUBMITTING -> ACCEPTED -> PROCESSING -> SUCCESS
```

**异常路径示例：**

```
SUBMITTING -> FAILED_RETRYABLE
PROCESSING -> FAILED_WAIT_COMPENSATE
PROCESSING -> FAILED_MANUAL
PROCESSING -> TIMEOUT
FAILED_RETRYABLE -> SUBMITTING
FAILED_WAIT_COMPENSATE -> PROCESSING
FAILED_MANUAL -> CLOSED
```

### 3.3 非法状态流转约束

以下流转默认禁止：
- `SUCCESS` -> 任意非终态
- `CANCELLED` -> `PROCESSING`
- `CLOSED` -> 任意其他状态
- 未经补偿确认的 `FAILED_WAIT_COMPENSATE` -> `SUCCESS`
- 无人工操作记录的 `FAILED_MANUAL` -> `SUCCESS`

若确有业务需要逆向修正，必须通过人工补录/纠偏流程执行，并形成审计记录，不得直接 update 状态字段覆盖历史。

### 3.4 任务终态判定规则

满足以下任一条件，可进入终态：
- 上游明确返回成功，且内部业务副作用已确认落账/落卡。
- 上游明确返回不可恢复失败，且无需继续补偿。
- 补偿流程完成，且补偿后的业务结果已确认。
- 人工确认关闭，并登记关闭原因、操作人、证据链接。

## 四、任务建模规范

### 4.1 必备字段

所有异步任务至少应包含以下字段：

| 字段 | 说明 |
|------|------|
| `task_no` | 内部任务号，全局唯一 |
| `task_type` | 任务类型，如 `CARD_OPEN` / `CARD_RECHARGE` / `CARD_WITHDRAW` |
| `biz_no` | 业务单号，如开卡申请单、充值单、转出单 |
| `merchant_id` / `user_id` | 业务归属主体 |
| `upstream_task_id` / `upstream_order_no` | 上游任务号或订单号 |
| `request_id` | 本次请求幂等键或链路请求号 |
| `status` | 当前任务状态 |
| `status_version` | 状态版本号，用于乐观锁控制 |
| `submit_count` | 提交次数 |
| `retry_count` | 重试次数 |
| `compensate_count` | 补偿次数 |
| `next_retry_at` | 下次重试时间 |
| `timeout_at` | 超时判定时间 |
| `last_error_code` | 最近错误码 |
| `last_error_message` | 最近错误摘要 |
| `result_snapshot` | 最近一次结果快照 |
| `created_at` / `updated_at` | 创建与更新时间 |
| `success_at` / `closed_at` | 成功/关闭时间 |

### 4.2 扩展字段要求

按任务类型可追加：
- **开卡：** `holder_id`、`card_bin_id`、`card_type`、`card_id`
- **充值：** `card_id`、`amount`、`currency`、`fee_amount`
- **转出：** `card_id`、`amount`、`currency`、`target_account_type`

### 4.3 子表与日志表要求

除任务主表外，建议至少补齐以下持久化对象：
- **任务状态流转日志表：** 记录每次状态迁移前后值、触发来源、操作者、原因。
- **任务重试记录表：** 记录每次自动重试/人工重试的触发条件、执行结果。
- **任务补偿记录表：** 记录补偿动作类型、前置原因、补偿结果、关联凭证。
- **任务事件关联表（可选）：** 关联 Webhook 事件 ID、轮询批次号、消息去重键。

## 五、轮询与回调协同规则

### 5.1 基本原则

轮询与回调不是二选一，而是协同关系：
- **回调优先推进：** 若已收到合法回调，应优先使用回调结果推进任务。
- **轮询负责兜底：** 若回调缺失、延迟、重复、签名异常或业务处理失败，轮询负责补确认。
- **统一判定落库：** 回调和轮询都不能绕开状态机直接改业务单状态。

### 5.2 推荐协同策略

- 提交上游后，任务进入 `ACCEPTED`/`PROCESSING`。
- 系统根据任务类型设定首轮轮询延时，例如 5s、15s、30s。
- **若回调先到：**
  - 先验签、去重、记录原始 payload
  - 调用状态机推进器尝试迁移状态
- **若轮询先到：**
  - 根据上游查询接口查询最新状态
  - 产出结果快照并尝试推进状态
- **若回调与轮询结果冲突：**
  - 不得直接覆盖成功结果
  - 进入冲突审计分支，优先保留证据并挂人工核查

### 5.3 优先级规则

默认优先级如下：
1. 已验签且字段完整的上游成功回调
2. 上游查询接口返回的成功结果
3. 已验签但信息不完整的回调（需补查）
4. 本地推断结果

**禁止**以本地推断覆盖上游明确失败/成功结果。

### 5.4 回调落库要求

所有 Webhook 事件必须：
- 原文入库到 `vcc_webhook_log` 或等价日志表
- 保存事件类型、签名、接收时间、处理结果、关联任务号
- 无论业务处理成功与否，都要保留接收证据

### 5.5 轮询终止条件

满足以下任一条件时停止自动轮询：
- 任务已进入成功终态
- 任务已进入明确关闭终态
- 已达到最大轮询次数并转入人工/补偿分支
- 上游明确返回无需继续查询

## 六、幂等与去重规范

### 6.1 发起幂等

业务发起时必须提供稳定幂等键，推荐优先级：
- 充值/转出：`biz_no` 或 `order_no`
- 开卡：申请单号 + `holder_id` + `bin_id` 的组合幂等键
- 对外请求：`request_id`

同一幂等键下：
- 若任务仍在处理中，返回已有任务，不重复发起
- 若任务已成功，返回既有成功结果摘要
- 若任务已失败且允许重试，应通过重试/补偿入口推进，不重新创建平行任务

### 6.2 回调去重

回调去重至少基于以下之一：
- 上游事件 ID
- `task_id`/`order_no` + `event_type` + `status`
- payload 指纹摘要

重复回调要求：
- 日志保留
- 业务处理短路
- 状态机不得重复写入同一迁移结果

### 6.3 轮询去重

轮询重复拿到相同状态结果时：
- 可以刷新最近查询时间
- 不应重复生成成功副作用
- 不应重复记账、重复发卡、重复扣款、重复补偿

### 6.4 副作用幂等

所有会产生业务副作用的动作必须具备幂等保护，包括但不限于：
- 账户扣款
- 卡内加款
- 转出记账
- 卡状态更新
- 站内通知/短信/邮件

建议通过"业务流水唯一键 + 状态版本 + 分布式锁/数据库唯一约束"组合保证。

## 七、失败分类与补偿策略

### 7.1 失败分类

首版将失败分为四类：
- **技术瞬时失败：** 网络抖动、上游超时、限流、临时不可用。
- **业务可恢复失败：** 上游处理中、回调缺失、结果未收敛、状态不一致待确认。
- **业务不可恢复失败：** 参数错误、余额不足、卡状态不允许、持卡人不存在。
- **人工介入失败：** 多源结果冲突、账务不平、重复扣款风险、补偿多次仍未收敛。

### 7.2 自动重试策略

仅以下失败允许系统自动重试：
- 网络超时
- 连接中断
- 查询接口偶发失败
- 明确的上游限流且可等待后重试

自动重试要求：
- 指数退避或阶梯退避，禁止高频打爆上游
- 达到阈值后必须转入 `FAILED_WAIT_COMPENSATE` 或 `FAILED_MANUAL`
- 每次重试都要有日志和结果记录

### 7.3 补偿动作分类

补偿动作可包括：
- 补查上游最终状态
- 重新触发查询任务
- 冻结待确认资金
- 冲正内部账务
- 标记卡状态待人工确认
- 通知运营/财务人工核对

### 7.4 补偿决策原则

- **先查实，再补偿：** 不能在未确认上游最终结果时盲目回滚。
- **资金补偿优先一致性：** 宁可短时间挂起，也不能做出双扣/双退。
- **卡状态补偿优先可追溯：** 例如开卡结果不明时，不得直接展示"开卡成功"。
- **补偿不可静默：** 所有补偿动作必须留痕。

### 7.5 人工介入触发条件

满足以下任一条件，应转人工：
- 回调成功但轮询失败，且多次补查仍冲突
- 资金已扣但上游订单结果未知
- 上游显示成功但内部落账失败
- 同一任务重试/补偿超过阈值
- 任务超时后仍无法确认最终账务归属

## 八、超时与兜底规则

### 8.1 超时分层

超时应区分三层：
- **请求超时：** 单次 HTTP 请求无响应
- **处理超时：** 上游已受理，但在约定窗口内未完成
- **业务闭环超时：** 任务长时间未收敛，影响账务或卡状态可信度

### 8.2 建议超时口径

首版建议值如下，后续可按上游 SLA 调整：

| 任务类型 | 请求超时 | 首次结果确认窗口 | 人工介入阈值 |
|---------|---------|----------------|------------|
| 开卡 | 10-15 秒 | 30 秒 | 10 分钟 |
| 充值 | 10-15 秒 | 1 分钟 | 15 分钟 |
| 转出 | 10-15 秒 | 1 分钟 | 15 分钟 |

> 说明：02.1 中已明确开卡"一般 30 秒内完成"；本规范以此作为首版默认口径。

### 8.3 超时后的标准处理

- **请求超时：** 不直接认定失败，应先查任务是否已受理
- **处理超时：** 进入 `TIMEOUT` 或 `FAILED_WAIT_COMPENSATE`，并触发补查
- **闭环超时：** 转 `FAILED_MANUAL`，生成人工处理单

### 8.4 兜底机制

系统必须至少具备以下兜底：
- 定时扫描长时间未收敛任务
- 批量补查上游终态
- 对资金异常任务发出告警
- 对人工待处理任务提供操作台或任务清单

## 九、三类典型任务说明

### 9.1 开卡任务

#### 9.1.1 任务特征

- 上游接口：`POST /rest/v1.0/vcc/card-open`
- 查询接口：`GET /rest/v1.0/vcc/card-open-task/{taskId}`
- 02.1 已定义返回 `taskId`，默认视为异步受理

#### 9.1.2 推荐状态路径

```
INIT -> SUBMITTING -> ACCEPTED -> PROCESSING -> SUCCESS
```

失败分支：
- 参数问题/持卡人不存在：直接失败关闭
- 开卡已受理但结果未明：进入补查/超时分支
- 开卡成功但卡激活失败：开卡任务成功，激活应拆成独立后续动作，不建议混在同一状态机里

#### 9.1.3 关键落库要求

- 保存上游 `taskId`
- 成功后回填 `cardId`、`card_no_mask`、卡状态
- 若需自动激活，应另建动作日志或独立任务，不覆盖开卡任务结果

### 9.2 充值任务

#### 9.2.1 任务特征

- 上游接口：`POST /rest/v1.0/vcc/card-recharge`
- 查询接口：`GET /rest/v1.0/vcc/card-recharge-query/{orderNo}`
- 02.1 已定义：部分 Bin 为同步模式，部分场景返回 `PENDING`

#### 9.2.2 推荐状态路径

- 同步成功：`INIT -> SUBMITTING -> SUCCESS`
- 异步确认：`INIT -> SUBMITTING -> ACCEPTED -> PROCESSING -> SUCCESS`
- 超时未确认：进入 `FAILED_WAIT_COMPENSATE`，禁止直接冲正，必须先补查上游结果

#### 9.2.3 关键风险点

- 资金已扣但卡未到账
- 上游成功但本地余额展示未刷新
- 重复请求导致双充值

#### 9.2.4 关键控制要求

- `orderNo` 必须全局唯一并作为强幂等键
- 账户扣款与充值任务状态推进要有一致性约束
- 对"本地失败、上游未知"的场景必须进入补偿队列

### 9.3 转出任务

#### 9.3.1 任务特征

- 上游接口：`POST /rest/v1.0/vcc/card-withdraw`
- 02.1 已说明流程同充值，支持同步/异步模式
- 属于资金敏感任务，审计要求不低于充值

#### 9.3.2 推荐状态路径

- 同步成功：`INIT -> SUBMITTING -> SUCCESS`
- 异步确认：`INIT -> SUBMITTING -> ACCEPTED -> PROCESSING -> SUCCESS`
- 结果未知：优先补查，不得直接把卡余额或账户余额当成成功依据

#### 9.3.3 补偿重点

- 防止重复转出
- 防止本地已记账、上游未扣减
- 防止上游已扣减、本地未回写
- 转出任务一旦出现账务不平，默认升级为人工介入优先级高的任务。

## 十、监控、审计与验收要求

### 10.1 监控要求

至少监控以下指标：
- 任务创建量、成功量、失败量、超时量
- 各任务类型平均处理时长、P95/P99 时长
- 回调接收量、验签失败量、重复回调量
- 自动重试次数、补偿次数、人工介入次数
- 长时间未收敛任务数

### 10.2 告警要求

以下场景必须告警：
- 某类任务连续失败率异常升高
- 回调大量失败或积压
- 超时任务数超过阈值
- 充值/转出账务不一致
- 同一任务发生异常状态跳变

### 10.3 审计要求

审计记录至少包括：
- 谁发起了任务
- 何时请求上游
- 上游返回了什么
- 哪次轮询/哪次回调推动了状态变化
- 何时进入补偿、由谁处理、结果如何
- 何时关闭任务、关闭依据是什么

资金类任务还应关联：
- 内部账务流水号
- 上游订单号/任务号
- 财务核对结果或人工备注

### 10.4 验收要求

后续实现与测试验收至少应覆盖：
- **正常成功链路：** 开卡、充值、转出各至少 1 条
- **上游异步成功链路：** 轮询成功、回调成功各 1 条
- **幂等链路：** 同一请求重复提交不得产生重复副作用
- **重复回调链路：** 不得重复入账、重复改状态
- **超时链路：** 任务超时后进入正确兜底分支
- **补偿链路：** 补查、人工介入、关闭流程可追溯
- **冲突链路：** 回调与轮询结果冲突时，不得静默覆盖

## 十一、与 02.1、02.5、03 的关联说明

### 11.1 与 02.1 的关联

02.1 已明确以下异步事实：
- 开卡通过 `taskId` 查询最终结果
- 充值/转出存在同步与异步并存模式
- 请求超时、系统错误等上游错误码已经定义

本规范在此基础上补齐：
- 内部统一状态机
- 超时与补偿口径
- 轮询/回调协同策略
- 幂等与副作用保护

### 11.2 与 02.5 的关联

- 02.5 负责"事件如何进入系统、如何验签、如何处理回调"。
- 02.6 负责"事件进入后，任务状态如何推进、哪些状态合法、何时补偿、何时转人工"。
- **一句话：02.5 管事件入口，02.6 管任务状态收敛。**

### 11.3 与 03 的关联

03 当前已有若干业务表和 `vcc_webhook_log` 基础能力，但异步任务统一建模尚未完全显式化。

后续若推进数据库补齐，应优先确保：
- 异步任务主表存在
- 状态流转日志存在
- 补偿记录存在
- 与充值、转出、卡片业务表可通过业务单号/任务号关联

## 十二、首版实施建议

1. 先以开卡、充值、转出三类任务统一落任务模型。
2. 以 `orderNo` / `taskId` / `biz_no` 为主轴打通幂等与查询链路。
3. Webhook 与轮询统一走状态机推进器，不允许各服务私自改终态。
4. 资金类任务先补审计与补偿，再谈吞吐优化。
5. 03 数据库设计文档应在下一轮补齐统一异步任务表结构。
