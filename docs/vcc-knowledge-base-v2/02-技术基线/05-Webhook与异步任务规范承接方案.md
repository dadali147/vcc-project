# 02.5-Webhook与异步任务规范承接方案

- **文档版本：** v1.0
- **更新时间：** 2026-03-22
- **状态：** 建议落地稿 / 待正式承接
- **适用范围：** VCC 02 区技术基线文档体系

## 一、方案结论

### 1.1 建议承接方式

建议拆文档，不建议继续深补到 02.1。

建议采用：
- **02.1-内部系统接口契约 v1**：继续保留为"接口总入口 / 能力索引 / 基础字段契约"
- **新增 02.5-Webhook 事件与验签规范**：承接所有回调入口、事件模型、签名校验、安全约束、应答语义
- **新增 02.6-异步任务状态机与补偿规范**：承接所有 task/order 异步处理模型、状态机、轮询/回调协同、幂等、重试、补偿、死信/人工介入规则

### 1.2 结论原因

当前 02.1 已具备以下定位：
- 已是正式入口文档，承接外部/内部接口清单与基础字段说明；
- 已包含 webhook 类型与异步接口的"存在性说明"；
- 但尚未展开验签规则、消费语义、幂等规则、状态推进、补偿边界；
- 这些内容属于运行时规范，不是简单接口字段补充；
- 若继续塞进 02.1，会导致一个文档同时承担：接口目录、字段字典、事件协议、任务编排、故障恢复，后续可维护性会明显下降。

因此，最稳妥的方案是：02.1 保持总入口，02.5/02.6 拆专项规范。

## 二、当前 02 区现状判断

### 2.1 已有文档角色

- **02.1-内部系统接口契约 v1**：已覆盖基础环境、认证、卡 Bin、持卡人、开卡、充值、交易查询、Webhook 类型概览、错误码、对接顺序；适合作为总入口。
- **04-业务流程文档**：已覆盖开卡、充值、交易、Webhook 回调类型、定时任务的业务级说明；适合作为流程侧背景，不适合承接低层技术约束。
- **06-三要素加密方案**：已承接敏感数据加解密专项；说明 02 区已经存在"总入口 + 专项技术文档"的组织方式。

### 2.2 缺口归类

当前缺口不是"少几个字段说明"，而是少以下运行规范：
- webhook 签名来源、签名串拼装、时间戳/重放保护；
- webhook 应答码语义、失败重试策略、消费确认时点；
- 授权类 webhook 的同步决策窗口与超时行为；
- 异步任务统一状态机；
- `taskId` / `orderNo` / `upstreamOrderNo` 关联关系；
- 幂等键、去重表、重复回调处理；
- 补偿任务触发条件、最大重试次数、人工介入条件；
- 轮询与 webhook 共存时的优先级与状态收敛规则。

这些内容更适合拆为专项文档。

## 四、02.5 建议文档设计

### 4.1 文档名

02.5-Webhook 事件与验签规范

### 4.2 文档目的

用于统一定义 VCC 系统对上游 webhook 的接入规则，避免开发、联调、测试阶段对"是否验签、何时确认、如何幂等、失败怎么重试"理解不一致。

### 4.3 建议章节结构

1. **文档范围与目标**
   - 本文适用的 webhook 类型
   - 与 02.1、04、06 的关系
   - 本文不负责的范围（业务决策本身、前端展示逻辑）

2. **Webhook 事件总览**
   - 事件类型总表
   - 每类事件的来源、触发场景、同步/异步属性
   - 是否需要业务判定
   - 建议最少覆盖：授权交易通知、转三方授权通知、卡片操作通知、OTP 通知、交易结果通知（如后续启用）

3. **回调入口契约**
   - URL 规划规则
   - HTTP Method
   - Content-Type
   - Header 约定
   - 超时阈值
   - 平台侧返回码定义

4. **验签与安全规范**
   - 签名字段来源
   - 签名原文拼装规则
   - 密钥来源与版本管理
   - 时间戳容忍窗口
   - 重放攻击防护
   - IP 白名单 / 网关限制 / HTTPS 强制要求
   - 验签失败处理策略

5. **事件模型与字段约束**
   - 公共字段
   - 事件唯一标识
   - `cardId` / `txnId` / `orderNo` / `upstreamOrderNo` 等关联字段定义
   - 必填 / 可选 / 空值处理规范
   - 枚举值映射规则

6. **应答语义规范**
   - 返回成功的定义
   - 返回失败的定义
   - 业务拒绝 vs 技术失败的区分
   - 哪些场景允许同步拒绝
   - 哪些场景只能异步处理后落状态

7. **幂等消费规范**
   - 幂等键推荐组合
   - 去重表/去重缓存要求
   - 重复事件处理结果
   - 已处理成功事件的重复应答策略

8. **失败重试与告警规范**
   - 上游重试预期
   - 我方失败分类（验签失败/参数非法/系统异常/业务未就绪）
   - 告警触发条件
   - 死信与人工处理入口

9. **各事件处理规则明细**

   建议每类事件统一模板：
   - 触发场景
   - 输入字段
   - 验签规则
   - 前置校验
   - 幂等键
   - 状态落库要求
   - 成功应答
   - 失败应答
   - 是否触发后续任务/补偿

10. **联调与验收清单**
    - 正常回调样例
    - 重复回调样例
    - 错签样例
    - 超时重试样例
    - 人工回放样例

### 4.4 验收标准

- 每类 webhook 都有明确入口、字段、应答语义；
- 验签规则可被开发直接实现，不存在"口头约定"；
- 幂等键已明确，重复回调处理结果已定义；
- 失败分类、重试、告警、人工介入规则已落文档；
- 至少附 1 组正常样例、1 组重复样例、1 组验签失败样例；
- 与 02.1、04 的引用关系清晰，不重复堆砌基础字段说明。

## 五、02.6 建议文档设计

### 5.1 文档名

02.6-异步任务状态机与补偿规范

### 5.2 文档目的

用于统一定义开卡、充值、转出等异步任务的状态推进、查询策略、回调协同、幂等、补偿、人工兜底规则，避免任务长期停留 `PROCESSING` / `PENDING` 而无统一收口。

### 5.3 建议章节结构

1. **文档范围与目标**
   - 适用任务范围：开卡、充值、转出、后续可扩展任务
   - 与 02.1、04、03 的关系

2. **异步任务总览**
   - 任务类型清单
   - 同步任务 / 异步任务划分
   - `taskId` / `orderNo` / `upstreamOrderNo` 关联关系图

3. **统一状态机定义**

   建议统一分层：
   - 受理态：`INIT` / `ACCEPTED`
   - 处理中：`PROCESSING` / `PENDING`
   - 成功态：`SUCCESS`
   - 失败态：`FAILED`
   - 终止态：`CANCELLED` / `CLOSED`（如适用）
   - 补偿态：`COMPENSATING` / `COMPENSATED` / `COMPENSATE_FAILED`（如采用）

   需明确：
   - 各状态进入条件
   - 各状态允许的前置状态
   - 非法状态跳转处理

4. **任务建模规范**
   - 主任务表最小字段集
   - 任务日志表/状态流转表建议
   - 外部请求号、内部任务号、业务单号映射
   - 错误码、错误原因、最近一次上游响应存储要求

5. **轮询与回调协同规则**
   - 什么时候轮询
   - 什么时候信任 webhook
   - 轮询与 webhook 同时到达时的收敛顺序
   - 最终一致性判定规则

6. **幂等与去重规范**
   - 创建任务幂等键
   - 状态推进幂等键
   - 重复查询/重复回调/重复补偿处理策略
   - 并发推进锁建议

7. **失败分类与补偿策略**

   建议分类：
   - 可重试技术失败
   - 不可重试业务失败
   - 上游未知态
   - 部分 / 本地失败

   每类必须定义：
   - 是否自动重试
   - 重试间隔
   - 最大次数
   - 是否进入人工处理
   - 是否触发资金/状态回滚

8. **超时与兜底规则**
   - `PROCESSING` 超时阈值
   - `PENDING` 超时阈值
   - 查询次数上限
   - 超时后的状态归并规则

9. **典型任务专项说明**

   建议最少展开：开卡任务状态机、充值任务状态机、转出任务状态机、交易结果异步确认（若纳入）

   每个任务建议按统一模板描述：
   - 创建入口
   - 唯一键
   - 上游响应
   - 状态推进
   - 查询策略
   - 回调协同
   - 失败补偿
   - 人工介入条件

10. **监控、审计与验收**
    - 待处理任务监控指标
    - 超时任务告警指标
    - 补偿失败告警指标
    - 人工处理台账要求

### 5.4 验收标准

- 至少覆盖开卡、充值、转出三类异步任务；
- 每类任务都有清晰状态机图或等价文本定义；
- 幂等键、查询规则、回调协同规则已明确；
- 超时、重试、补偿、人工介入阈值已明确；
- 与数据库设计文档能形成字段映射；
- QA 可基于文档直接设计状态流转测试用例。
