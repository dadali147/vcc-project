# VCC 业务流程文档

VCC 业务流程文档

本文档描述 VCC 虚拟卡发卡台的核心业务流程，包括开卡、充值、交易等关键环节。

## 一、卡类型定义

| 类型 | 名称 | 资金模式 | 适用场景 |
|------|------|---------|---------|
| PREPAID | 储值卡 | 用户账户 → 卡内余额 → 消费 | 个人用户、小额消费 |
| BUDGET | 预算卡/共享卡 | 用户账户设置额度 → 消费时直接扣账户 | 企业用户、资金池管理 |

**共享卡说明**：共享卡不是与他人共用一张卡，而是用户自己多张卡共享一个资金池。用户可以创建一个预算金额（资金组），自己名下的所有卡片都可以共用该余额，无需单卡单充值。一个资金组也可以只绑定一张卡。

### 费用扣除规则
| 费用类型 | 储值卡 | 预算卡 |
|---------|--------|--------|
| 开卡费 | 用户账户扣除 | 用户账户扣除 |
| 月费 | 用户账户扣除 | 用户账户扣除 |
| 交易手续费 | 卡内余额扣除 | 用户账户扣除 |
| 小额手续费 | 卡内余额扣除 | 用户账户扣除 |
| 跨境交易费 | 卡内余额扣除 | 用户账户扣除 |

## 二、开卡流程

### 2.1 流程概览
创建持卡人 → 申请开卡 → 查询开卡结果 → 激活卡片 → 获取三要素

### 2.2 详细步骤

**Step 1: 创建持卡人**
- 触发: 用户首次开卡或需要新增持卡人时
- 操作: 用户填写持卡人信息（姓名、手机号、邮箱等）
- 系统处理:
  - 校验信息完整性
  - 调用上游创建持卡人接口
  - 保存持卡人信息到 vcc_card_holder 表
- 异常处理: 上游返回错误时提示用户修改信息

**Step 2: 申请开卡**
- 触发: 用户选择持卡人、卡类型、BIN后提交申请
- 前置检查:
  - 用户账户余额 ≥ 开卡费
  - 持卡人状态正常
  - 该BIN支持所选卡类型
- 系统处理:
  - 扣除开卡费（用户账户）
  - 调用上游开卡申请接口（异步）
  - 创建卡记录，状态为 APPLYING
  - 生成开卡申请流水号

**Step 3: 查询开卡结果**
- 触发: 定时任务轮询 / 用户主动查询
- 时间窗口: 开卡申请后30秒内
- 系统处理:
  - 调用上游查询接口
  - 成功: 更新卡状态为 ACTIVE 或 PENDING_ACTIVATION
  - 失败: 更新卡状态为 FAILED，退还开卡费
  - 处理中: 保持 APPLYING 状态，继续轮询

**Step 4: 激活卡片（如需要）**
- 触发: 卡状态为 PENDING_ACTIVATION
- 系统处理:
  - 调用上游激活接口
  - 成功: 更新状态为 ACTIVE
  - 失败: 记录错误日志，进入人工处理队列

**Step 5: 获取三要素**
- 触发: 用户查看卡片详情时
- 安全机制:
  - 三要素（卡号、CVV、有效期）AES加密存储在上游
  - 系统调用接口获取加密数据
  - 前端使用密钥解密展示（服务端不存储明文）
- 审计: 记录三要素获取日志

### 2.3 状态流转图
```
APPLYING(申请中) → PENDING_ACTIVATION(待激活) → ACTIVE(正常)
                              ↓
                         ACTIVE(正常)
                              ↓
              ┌───────────────┼───────────────┐
              ↓               ↓               ↓
          FROZEN(冻结)   SUSPENDED(停用)   CLOSED(已销卡)
              ↓               ↓
          ACTIVE(解冻)    CLOSED(销卡)
```

## 三、充值流程

### 3.1 USDT充值到账户
用户发起充值 → 生成TRC20地址 → 链上监控 → 到账确认 → 账户余额增加

| 步骤 | 说明 |
|------|------|
| 发起充值 | 用户选择充值金额，系统显示TRC20地址和二维码 |
| 地址分配 | 系统分配或生成唯一的TRC20充值地址（可复用） |
| 链上监控 | TronUsdtChecker定时扫描链上交易 |
| 到账确认 | 检测到入账后，根据确认数（通常6-19个）确认到账 |
| 入账处理 | 扣除入账费率后，增加用户账户余额 |
| 通知 | 发送邮件/站内信通知用户充值成功 |

### 3.2 账户充值到卡内（储值卡）
用户选择卡片和金额 → 系统检查账户余额 → 扣除账户余额 → 调用上游充值 → 查询结果

| 步骤 | 说明 |
|------|------|
| 发起充值 | 用户选择要充值的卡片和金额 |
| 余额检查 | 检查用户账户余额 ≥ 充值金额 + 卡充值费率 |
| 费用扣除 | 从用户账户扣除充值金额 + 手续费 |
| 上游调用 | 调用YeeVCC充值接口（异步） |
| 状态查询 | 定时查询充值结果（30秒内） |
| 结果处理 | 成功: 卡余额增加；失败: 退还账户 |

### 3.3 预算卡额度设置
用户选择卡片和额度 → 系统检查账户余额 → 设置额度上限 → 完成
- 预算卡不实际充值到卡内，而是设置消费额度上限
- 消费时直接从用户账户扣款
- 可随时调整额度（上调需检查账户余额）

## 四、交易流程

### 4.1 授权交易通知（Webhook）
商户发起扣款 → 上游授权 → Webhook通知 → 系统处理 → 返回结果

| 步骤 | 说明 |
|------|------|
| 商户扣款 | 绑卡商户发起扣款请求 |
| 上游授权 | YeeVCC进行风控检查和余额校验 |
| 通知下发 | YeeVCC发送授权交易通知Webhook |
| 系统处理 | 验证签名 → 查找卡片 → 检查余额 → 记录交易 |
| 转授权决策 | 根据风控规则决定是否通过 |
| 返回结果 | 回调上游告知处理结果 |

### 4.2 交易费用扣除
储值卡交易:
- 交易金额从卡内扣除
- 各类手续费（小额、跨境等）从卡内扣除
- 卡余额可为负数（记录欠费）

预算卡交易:
- 交易金额从用户账户扣除
- 各类手续费从用户账户扣除
- 检查账户余额是否充足

### 4.3 交易状态流转
```
PENDING(待处理) → AUTHORIZED(已授权) → CAPTURED(已扣款)
       ↓
   DECLINED(拒绝)
       ↓
   REFUNDED(已退款)
       ↓
   CANCELLED(已撤销)
```

## 五、销卡流程

### 5.1 主动销卡
用户申请销卡 → 检查卡状态 → 调用上游销卡 → 退还余额 → 完成

| 步骤 | 说明 |
|------|------|
| 申请销卡 | 用户在界面发起销卡申请 |
| 状态检查 | 检查卡片无未完成的交易、无欠费 |
| 上游调用 | 调用YeeVCC销卡接口 |
| 余额处理 | 储值卡: 退还卡内余额到用户账户 |
| 状态更新 | 更新卡状态为 CLOSED |

### 5.2 异常销卡
- 持卡人信息异常时，上游可能主动销卡
- 通过Webhook通知系统
- 系统更新卡状态，通知用户

## 六、Webhook回调处理

### 6.1 回调类型
| 回调类型 | 说明 | 处理逻辑 |
|---------|------|---------|
| 授权交易通知 | 商户发起扣款时的授权请求 | 风控检查 → 转授权决策 |
| 转三方授权 | 需要用户确认的授权 | 发送OTP/通知用户 |
| 卡片操作通知 | 冻结/解冻/销卡等操作通知 | 同步卡片状态 |
| OTP通知 | 需要验证码的交易 | 转发OTP给用户 |
| 交易结果通知 | 交易最终状态 | 更新交易记录 |

### 6.2 回调处理流程
接收Webhook → 验签 → 解析 → 业务处理 → 记录日志 → 返回200
- 所有回调原始数据存入 vcc_webhook_log
- 验签失败记录告警
- 业务处理异常进入重试队列

## 七、定时任务
| 任务名称 | 频率 | 说明 |
|---------|------|------|
| 开卡结果查询 | 每30秒 | 轮询申请中的开卡请求 |
| 充值结果查询 | 每30秒 | 轮询处理中的充值请求 |
| 余额同步 | 每小时 | 同步卡片余额 |
| 交易同步 | 每小时 | 同步缺失的交易记录 |
| USDT到账检查 | 每2分钟 | 扫描TRC20链上交易 |
| 预算卡额度同步 | 每天 | 同步预算卡消费额度 |

## 八、异常处理

### 8.1 开卡失败
- 退还开卡费到用户账户
- 记录失败原因
- 允许用户重新申请

### 8.2 充值失败
- 退还充值金额和手续费到用户账户
- 记录失败原因
- 通知用户重新尝试

### 8.3 交易争议
- 记录争议交易
- 冻结相关资金
- 进入人工审核流程
