# VCC 重构方案文档

VCC 虚拟卡发卡台 - 重构方案文档
版本：v0.3 | 状态：草稿 | 最后更新：2026-03-13
本文档记录 VCC 虚拟卡发卡台重构项目的整体方案，包括架构设计、功能调整清单、开发计划等。

## 一、项目背景

### 1.1 重构原因
- 旧系统对接多家上游（Qbit / 华人 / YouCard），各上游适配代码直接耦合在业务层，改一个地方到处出问题
- 大量 JdbcTemplate 裸 SQL，MyBatis 形同虚设
- 命名混乱（如 lizi.java），代码可读性差，维护成本高
- 新增上游需要全局改动，不可持续

### 1.2 重构目标
- 彻底隔离上游适配层，业务逻辑与上游解耦
- 代码结构清晰，模块职责单一
- 只对接一家新上游（YeeVCC），边界清晰
- 为后续多上游扩展预留接口

### 1.3 新上游信息
- 上游名称：YeeVCC（Yeewallex）
- API 文档：https://open.kun.global/docs/apis/yeevcc/get__rest__v1.0__vcc__user-info
- 测试环境：https://apiqa.kun.global/yop-center
- 生产环境：https://api.kun.global/yop-center
- 认证方式：RSA2048 + AES16 加密

### 2.2 模块划分
```

├── vcc-admin          # 启动入口

├── vcc-system         # 系统管理（用户 / 角色 / 权限）

    │   └── yeevcc/YeeVccGatewayImpl.java # YeeVCC 实现
    ├── domain/        # 业务实体
    ├── service/       # 业务逻辑
    ├── controller/    # API 接口
    ├── webhook/       # 回调处理
    └── task/          # 定时任务
```

### 2.3 上游适配层接口设计（IUpstreamGateway）
所有业务逻辑只调用此接口，不感知具体上游：
- 持卡人管理：添加持卡人
- 开卡流程：申请开卡、查询开卡结果、激活卡片
- 卡片信息：获取三要素（卡号/CVV/有效期，AES 加密）、查询余额
- 卡片操作：冻结、解冻、销卡
- 资金操作：充值、查询充值结果、转出
- 交易查询：获取交易记录
- 配置查询：获取可用 BIN 列表

### 4.9 费率精细化配置（✨ 新增）
基于业务需求，新版系统支持多维度、精细化费率配置：

| 费率类型 | 说明 | 扣费方式 |
|---------|------|---------|
| 入账费率 | 用户充值到账户的手续费 | 从充值金额扣除 |
| 卡充值费率 | 从账户充值到卡的手续费 | 从账户余额扣除 |
| 小额手续费 | 小额交易额外费用 | 卡内扣除 |
| 跨境交易手续费 | 跨境交易额外费用 | 卡内扣除 |
| 单笔交易费 | 每笔交易固定费用 | 卡内扣除 |
| 零元验证费 | 0元验证交易费用 | 卡内扣除 |
| 失败交易费 | 交易失败费用 | 卡内扣除 |
| 退款交易费 | 退款处理费用 | 卡内扣除 |
| 撤销费 | 交易撤销费用 | 卡内扣除 |

扣费逻辑说明：
- 部分费用与上游同步收取（如入账费率）
- 部分费用为平台自行卡内扣除（如小额手续费、跨境费等）
- 卡内扣除的费用需额外记录，并进行卡内划转
- 卡余额可为负数：当卡内余额不足以划转时，记录欠费金额，用户看到的卡余额为计算后的负数（非真实负数）

## 七、技术评估建议

### 7.1 数据库选型（MySQL vs PostgreSQL）
| 维度 | MySQL 8.0 | PostgreSQL |
|------|-----------|------------|
| 性能 | 读写优秀，适合 OLTP | 复杂查询更优 |
| JSON 支持 | 基础支持 | JSONB 类型，索引完善 |
| 分区表 | 支持 | 更灵活 |
| 团队熟悉度 | 高 | 需学习成本 |
| 迁移成本 | 低 | 中等 |

建议：如当前 MySQL 无性能瓶颈，建议保持 MySQL；如有特定需求（如复杂 JSON 查询），再考虑 PG。

### 7.2 三要素加密策略
| 方案 | 安全性 | 性能 | 用户体验 | 建议 |
|------|--------|------|---------|------|
| 服务端加密 | 中 | 高 | 好 | ⭐ 推荐 |
| 客户端加密 | 高 | 中（批量受影响） | 一般 | 高安全场景 |
| 混合模式 | 高 | 按需 | 灵活 | 可选策略 |

建议：默认服务端加密，高安全要求用户可选客户端加密。

### 7.3 部署方案

方案 A：Docker + Docker Compose（推荐）
- 环境一致性高
- 易于本地开发和测试
- 可逐步过渡到 K8s

方案 B：传统部署
- 团队熟悉度高
- 运维简单
- 适合初期快速上线
