# 异常与边界场景基线 v1

## 1. 文档定位

本文件用于统一 VCC 项目中关键业务的失败态、处理中、部分成功、回调延迟、重复提交、冻结状态、余额不足等边界场景口径，避免前后端、接口、测试各自解释。

## 2. 覆盖范围

- 开卡
- 充值
- 交易
- 销卡
- 导出
- KYC

## 3. 通用边界原则

- 成功、处理中、失败必须明确区分，不允许用模糊状态替代
- 异步场景必须定义"用户可见反馈"与"最终状态来源"
- 重复提交必须有幂等或防重口径
- 阻塞/失败场景必须可追溯到原因与责任对象

## 4. 模块级边界场景

### 4.1 开卡

| 场景 | 处理规则 |
|------|---------|
| 余额不足 | 不可进入成功态，应明确提示费用不足 |
| 上游处理中 | 前端展示处理中，最终结果以后续状态或回调为准 |
| 重复提交 | 同一请求不得生成多张卡 |
| 参数不完整 | 直接失败，不进入处理中 |

### 4.2 充值

| 场景 | 处理规则 |
|------|---------|
| 充值处理中 | 展示处理中，不提前增加可用余额 |
| 充值失败 | 不得口头算到账，必须保留失败记录 |
| 重复提交 | 同一充值请求不得重复记账 |
| 回调延迟 | 前端展示处理中，并提示结果待确认 |

### 4.3 交易

| 场景 | 处理规则 |
|------|---------|
| 余额不足 | 直接失败或拒绝，状态不可混同为处理中 |
| 冻结中 | 应区分冻结成功、结算成功、释放失败等阶段 |
| 退款处理中 | 应与消费成功状态分离展示 |
| 拒付/撤销 | 后续版本需继续细化口径 |

### 4.4 销卡

| 场景 | 处理规则 |
|------|---------|
| 卡内有余额 | 必须先按正式规则处理资金去向 |
| 存在未结清交易 | 不得直接视为完全销卡完成 |
| 冻结中卡片 | 需先明确是否允许销卡及处理顺序 |

### 4.5 导出

| 场景 | 处理规则 |
|------|---------|
| 任务创建成功 | 不等于文件已生成 |
| 文件生成中 | 应展示处理中 |
| 文件失败 | 应允许查看失败原因或重试 |
| 重复发起 | 应有频控或任务复用口径 |

### 4.6 KYC

| 场景 | 处理规则 |
|------|---------|
| 提交成功 | 不等于审核通过 |
| 审核中 | 应明确展示处理中 |
| 审核拒绝 | 需展示拒绝结果及后续动作口径 |
| 重复提交 | 需定义覆盖、追加还是禁止提交 |

## 5. 用户可见反馈原则

- 用户前端只展示统一业务口径，不展示内部技术异常细节
- 处理中状态必须告诉用户"结果待确认"
- 失败状态应尽量给出可执行下一步，而不是只显示失败

## 6. 验收标准

- 覆盖开卡、充值、交易、销卡、导出、KYC 六大模块
- 每个模块至少包含触发条件、系统处理、前端展示、结果状态
- 明确处理中与失败的边界
- 可作为页面状态、接口状态和测试用例的统一依据
