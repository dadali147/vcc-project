## 第三轮补齐结果（2026-03-22）

### 一、状态字典逐项映射记录（第一版）

> 说明：当前《01.3-业务规则-状态字典 v1》仍是骨架，尚未形成正式枚举全集。以下内容先作为《02.1-内部系统接口契约 v1》的交叉校验映射底稿，用于先收口接口侧状态口径，后续再反写状态字典正式版。

#### 1. registerStatus 映射

当前 02.1 已出现值：`REGISTERED`、`PENDING_REVIEW`、`REJECTED`、`ACTIVE`

**当前业务语义拆解：**
- `REGISTERED`：注册资料已提交且基础账户已创建，但尚未进入最终可用态
- `PENDING_REVIEW`：注册后仍需人工审核 / 邮件激活 / 补资料时的处理中态
- `REJECTED`：注册申请被拒绝，不可进入后续 KYC / 开卡链路
- `ACTIVE`：注册流程已完成，可正常进入后续业务操作

**页面/流程映射：**
- 注册提交成功页：优先展示 `REGISTERED` 或 `PENDING_REVIEW`
- 登录前置校验：若为 `REJECTED` / 非 `ACTIVE`，需返回明确阻断原因
- 自动登录场景：若注册即激活，可直接返回 `ACTIVE`

**当前问题：**
- `REGISTERED` 与 `PENDING_REVIEW` 边界未完全钉死
- 尚未定义是否存在 `INIT` / `EMAIL_VERIFIED` / `DISABLED` 等中间态

**结论：** 接口契约可先保留以上 4 个值，但状态字典正式版必须补"状态进入条件 / 可跳转路径 / 前端文案"三列

#### 2. kycStatus 映射

当前 02.1 已出现值：`SUBMITTED`、`UNDER_REVIEW`、`REJECTED`、`APPROVED`

**当前业务语义拆解：**
- `SUBMITTED`：KYC 资料已提交，系统已受理
- `UNDER_REVIEW`：进入自动校验或人工审核流程中
- `REJECTED`：审核拒绝，需要补件或重新提交
- `APPROVED`：KYC 审核通过，可继续开卡等后续能力

**页面/流程映射：**
- KYC 提交成功页：优先显示 `SUBMITTED` / `UNDER_REVIEW`
- KYC 结果页：展示 `REJECTED` 的 `rejectReason`，或 `APPROVED` 的完成态
- 登录后首页 / 顶部提示：根据 `kycStatus` 决定是否引导去开卡

**当前问题：**
- `SUBMITTED` 与 `UNDER_REVIEW` 是否都要保留还需产品定稿
- 尚未定义是否存在 `INIT` / `EXPIRED` / `MORE_INFO_REQUIRED`

**结论：** 接口侧先按 4 态收口；若后续出现补件场景，建议新增 `MORE_INFO_REQUIRED`，不要复用 `REJECTED`

#### 3. cardStatus 映射

当前 02.1 已出现 / 暗含值：冻结、解冻、销卡、未激活等语义；错误码出现"已冻结""已注销""未激活"

**当前建议统一状态集（接口侧第一版）：**
- `PENDING_ACTIVATION`：卡已生成但未激活 / 未可用
- `ACTIVE`：正常可用
- `FROZEN`：已冻结，不可消费
- `CLOSED`：已销卡 / 已注销，不可恢复使用

**页面/流程映射：**
- 卡片列表：状态筛选、状态标签展示统一使用上述值
- 卡片详情：冻结/解冻/销卡按钮显隐与 `cardStatus` 强绑定
- 敏感信息查看 / OTP：建议仅 `ACTIVE` 卡默认可用，其它状态按业务规则限制

**当前问题：**
- 02.1 还未把 `cardStatus` 正式枚举收口为常量列表
- "未激活"当前只是错误语义，还未写成正式枚举名
- 尚未定义是否需要 `APPLYING`、`ACTIVATING`、`EXPIRED`

**结论：** 本轮先收口 `PENDING_ACTIVATION` / `ACTIVE` / `FROZEN` / `CLOSED`；后续由上游回调和数据库设计再决定是否拆更细状态

#### 4. applyStatus 映射

当前 02.1 已出现值：`PROCESSING`；异步任务结构已定义 `businessStatus`

**当前建议统一开卡申请语义：**
- `SUBMITTED`：申请已提交，等待系统处理
- `PROCESSING`：处理中，已进入上游或内部执行链路
- `SUCCESS`：开卡成功，已产出 `cardId`
- `REJECTED`：申请被业务拒绝
- `FAILED`：系统执行失败或上游失败

**页面/流程映射：**
- 开卡页面提交后：先展示 `SUBMITTED` / `PROCESSING`
- 开卡记录页：状态列统一使用 `applyStatus`
- 异步任务查询：`taskStatus` 表示任务执行态，`applyStatus` 表示开卡业务态，不能混用

**当前问题：**
- 02.1 示例只写了 `PROCESSING`，成功/拒绝/失败未在正文逐项说明
- `applyStatus` 与 `taskStatus` 边界容易被前端写乱

**结论：** 开卡相关页面必须同时保留 `taskStatus` + `applyStatus` 双层语义，不能只返回一个状态字段糊弄

#### 5. taskStatus 映射

当前 02.1 已定义值：`INIT`、`PROCESSING`、`SUCCESS`、`FAILED`、`PARTIAL_SUCCESS`、`CANCELED`、`EXPIRED`

**当前业务语义拆解：**
- `INIT`：任务已创建未执行
- `PROCESSING`：处理中
- `SUCCESS`：任务执行完成
- `FAILED`：任务执行失败
- `PARTIAL_SUCCESS`：批量任务部分成功
- `CANCELED`：任务已取消
- `EXPIRED`：结果已失效 / 下载地址过期

**页面/流程映射：**
- 开卡结果查询、充值结果查询、对账单导出查询统一复用
- 导出类页面对 `EXPIRED` 必须提示重发任务
- `SUCCESS` 只代表任务执行完成，不等于业务成功，仍需看 `businessStatus`

**当前问题：**
- 目前只是异步任务章节有定义，尚未扩散到具体页面字段口径

**结论：** `taskStatus` 是目前已收口最完整的状态集，可反向作为状态字典正式版模板

#### 6. budgetStatus 映射

当前 02.1 已出现语义：预算设置、预算调整、`availableBudget`、预算调整审计；但 `budgetStatus` 未正式定义

**当前建议统一预算状态集（接口侧第一版）：**
- `EFFECTIVE`：预算已生效，可正常使用
- `PENDING_EFFECTIVE`：预算已提交，等待生效时间到达
- `EXHAUSTED`：预算已用尽
- `DISABLED`：预算已停用 / 已失效

**页面/流程映射：**
- 共享卡限额页：状态列使用 `budgetStatus`
- 卡片详情预算区：预算剩余额度展示需结合 `budgetStatus`
- 调额记录页：记录调额动作本身，不等于预算当前态

**当前问题：**
- 预算卡与储值卡语义不同，预算状态只适用于 `BUDGET` 卡，不可套到 `PREPAID`
- 目前 02.1 只写了预算金额和记录，没有正式 `budgetStatus` 字段定义

**结论：** 预算状态需要在 02.1 预算章节显式增加字段，不然前端只能自己造词，后面一定翻车

**剩余问题：**

### 三、数据库设计对齐记录（第一版草稿）

> 说明：本轮未直接读取数据库设计正文，以下仅基于 02.1 当前接口契约先做"应落库对象与关键字段"预对齐，作为下一轮与 03-数据库设计文档逐项核对的底稿，不能视为最终数据库结论。

#### 1. 账户与注册域

- 接口涉及核心字段：`userId`、`merchantId`、`email`、`registerStatus`、`inviteCode`
- 预判应有落库对象：商户主账户表、商户主体表、注册审核/归因记录表
- 需核数据库设计点：
  - `registerStatus` 存主账户表还是单独审核表
  - `inviteCode` / 渠道归因是否有独立字段
