# 03-开发与变更唯一基准清单

## 一、文档定位

本文件是 VCC 重构项目在“开发执行、子 Agent 派单、业务变动处理”场景下的**统一引用入口**。

目的只有一个：
**把后续所有开发动作的引用顺序定死，避免不同人各自引用旧稿、聊天结论、临时截图、口头理解，最终把项目做偏。**

只要涉及以下事项，必须优先回到本清单确认正式基线：
- 新需求拆解
- 子 Agent 正式派单
- 业务规则调整
- 接口新增 / 修改
- 数据库结构调整
- 页面字段与交互变更
- 测试、验收、回归、发布前核对

---

## 二、给谁看

本清单不是写给“旁观者”看的，是写给所有会实际推进项目的人看的。

### 适用对象
1. **主 Agent / Agent 0**
   - 负责需求拆解、派单、架构判断、验收归口
   - 必须用本清单决定“先看什么、引用什么、什么能派、什么不能派”

2. **业务开发 Agent**
   - 包括后端、前端、Webhook、测试、QA、文档治理等子 Agent
   - 接单前必须先按本清单回读正式基线，不能凭历史上下文直接开工

3. **产品、项目管理、验收角色**
   - 在确认需求、判断是否偏题、检查结果物时，必须回到本清单对应文档核对

4. **后续接手人员 / 新加入成员**
   - 用本清单快速定位“当前项目到底以什么为准”，避免把旧资料当现行标准

---

## 三、要解决什么问题

本清单重点解决以下几个老问题：

1. **引用口径混乱**
   - 同一个业务问题，有人看旧派单稿，有人看聊天记录，有人看代码现状，有人看脑内理解。
   - 结果就是：开发口径不一致，验收也无从判断。

2. **旧资料替代正式基线**
   - 历史草稿、临时纪要、截图、群聊结论经常被误当成正式标准。
   - 这类材料可以辅助理解，但**不能替代正式基线**。

3. **先开发、后补文档**
   - 一旦默认“先做再说”，项目会越来越依赖口头记忆和代码现状，最后文档失真、派单失真、验收失真。

4. **变更没有回写基线**
   - 业务已经变了，但正式文档没更新，后面所有 Agent 继续引用旧标准，错误会被持续放大。

一句话：
**本清单就是为了把“开发依据”统一成一套正式基线，任何执行动作都不能绕开它。**

---

## 四、核心基线列表

以下内容构成后续开发与变更处理时的**核心正式基线**。凡是正式派单、开发实现、联调、测试、验收，必须优先引用这些基线文档，而不是引用散落资料。

### 1. 业务总纲 / 重构方案
**作用**：定义项目范围、核心目标、模块边界、整体重构方向。  
**回答的问题**：这次重构到底在做什么、不做什么、整体边界是什么。  
**使用场景**：任务拆解、模块归属判断、是否属于本期范围判断。  
**优先级**：最高层总纲基线。

### 2. 业务流程
**作用**：定义开卡、充值、消费、退款、状态流转等关键业务链路。  
**回答的问题**：业务如何流转、前后步骤是什么、不同节点谁触发什么动作。  
**使用场景**：流程实现、联调判断、异常流梳理、测试用例设计。

### 3. 业务规则 / 状态字典
**作用**：定义状态含义、状态迁移条件、规则口径、费用与限制条件。  
**回答的问题**：什么状态合法、什么条件能变更、规则如何解释。  
**使用场景**：后端规则实现、前端状态展示、测试断言、异常判断。

### 4. 页面基线 / 页面清单与字段
**作用**：定义页面范围、页面职责、字段口径、展示与操作入口。  
**回答的问题**：页面应该有什么、字段应该怎么理解、哪些操作属于哪个页面。  
**使用场景**：前端开发、管理端/商户端字段对齐、验收页面缺漏。

### 5. 内部系统接口契约
**作用**：定义内部接口的入参、出参、字段含义、错误码、调用关系。  
**回答的问题**：系统之间怎么对接、接口字段以什么为准、返回结构如何统一。  
**使用场景**：后端开发、前后端联调、接口测试、Mock、验收。

### 6. Webhook / 异步任务规范
**作用**：定义回调、异步通知、补偿、重试、幂等、事件处理边界。  
**回答的问题**：异步事件怎么进入系统、失败如何处理、重复回调怎么防。  
**使用场景**：Webhook 开发、事件流设计、对账与补偿设计、回归测试。

### 7. 数据库 DDL 正式基线
**作用**：定义正式表结构、字段设计、索引、约束、状态存储口径。  
**回答的问题**：数据库应该长什么样、字段是否存在、字段语义是什么。  
**使用场景**：后端持久层实现、SQL 编写、迁移脚本、数据校验、验收。

## 五、正式引用顺序

以后所有开发、派单、变更、验收，默认按下面顺序引用，不允许跳着来。

### 场景一：新任务 / 新派单
1. **先看业务总纲 / 重构方案**
   - 先判断任务是否属于当前范围，归属哪个模块。
2. **再看业务流程**
   - 确认业务链路和上下游动作。
3. **再看业务规则 / 状态字典**
   - 确认规则条件、状态含义、限制口径。
4. **按需补看页面基线 / 接口契约 / Webhook 规范 / 数据库 DDL**
   - 前端任务重点看页面基线
   - 接口任务重点看接口契约
   - 回调任务重点看 Webhook 规范
   - 持久化任务重点看数据库 DDL

### 场景二：业务变更
1. 先确认变更影响的是**哪一层基线**  
2. 先更新对应正式基线文档  
3. 再更新关联基线文档  
4. 最后才能重新派单、开发、联调、验收

### 场景三：结果验收
1. 对照业务总纲确认任务是否偏题  
2. 对照业务流程确认流程是否打通  
3. 对照状态字典确认规则是否落准  
4. 对照页面 / 接口 / Webhook / DDL 确认实现细节是否一致

## 六、哪些内容禁止替代正式基线

下面这些内容可以作为辅助材料，但**一律不得替代本清单对应的正式基线**：

1. **历史旧文档**
   - 包括旧目录下未归位文档、旧版本方案、过期设计稿。

2. **临时派单稿 / 草稿 / 讨论稿**
   - 派单稿的作用是组织执行，不是天然正式业务标准。

3. **聊天记录 / 会议纪要 / 口头结论**
   - 即使结论正确，也必须回写正式基线后才能生效。

4. **截图 / 白板 / 脑图 / 表格片段**
   - 这些只能辅助说明，不能作为正式实施依据。

5. **代码现状**
   - 代码只能说明“当前做成了什么”，不能天然代表“应该做成什么”。
   - 若代码与正式基线冲突，以正式基线为准；需要调整时先记录冲突，再按流程处理。

6. **个人理解 / 口头传达 / 上下文记忆**
   - 不管是谁说的，没有回写正式基线，就不算正式口径。

一句难听但必须说的话：
**谁拿旧稿、聊天、截图、代码现状去替代正式基线，谁就是在给后续返工埋雷。**

---

## 七、禁止事项

为防止再次出现“口径漂移”和“虚假执行”，以下事项明确禁止：

1. **禁止未读基线直接派单**
   - 没有按本清单核对正式基线，不允许进入正式派单。

2. **禁止未更新基线先改业务**
   - 业务规则、接口字段、状态口径、数据库结构发生变化时，必须先改基线，再进开发。

3. **禁止拿旧资料当现行标准**
   - 发现旧资料与正式基线冲突时，旧资料只能归档或标注失效，不能继续引用。

4. **禁止用“代码已经这样了”反推业务规则**
   - 代码现状不是正式规则来源。

5. **禁止多版本并行口径**
   - 同一时期只能有一套当前有效的正式基线。
   - 如需切换版本，必须明确版本状态、适用范围、生效时点。

6. **禁止跳过变更回写**
   - 任何业务变更若未回写正式基线，不得对外宣称“已经确认”或“可以开发”。

---

## 八、变更时操作原则

当出现新需求、业务调整、技术方案变动、字段改动、流程补充时，统一按以下原则执行：

### 原则 1：先定基线，再动执行
凡是会影响开发判断的内容，必须先落到正式基线文档，再启动或继续执行。

### 原则 2：谁改规则，谁推动回写
谁提出变更，谁负责推动对应基线更新；不能只在聊天里说一句“按新的来”。

### 原则 3：先改单点基线，再校验关联基线
例如接口字段变了，不只是改接口契约；还要检查是否同步影响：
- 页面字段
- 状态字典
- 数据库字段
- Webhook / 异步处理
- 测试与验收口径

### 原则 4：冲突先记录，不能拍脑袋合并
如果正式基线之间出现冲突，或基线与代码现状冲突：
- 先记录冲突点
- 明确由谁裁决
- 裁决后统一回写
- 未裁决前不得各自按理解继续扩散执行

### 原则 5：变更完成后再重新派单
变更一旦成立，必须使用更新后的基线重新组织后续派单、开发、联调和验收，不允许沿用旧任务口径继续推进。

---

## 九、执行口径

从本文件生效起，后续所有正式开发动作默认遵循以下执行口径：

- **先基线，后派单**
- **先文档，后开发**
- **先更新，后变更**
- **先核对，后验收**

如果某项内容在正式基线中还不存在，正确动作不是“先按理解开发”，而是：
1. 先补齐基线；
2. 明确归属；
3. 再进入执行。

---

## 十、结论

本清单是 VCC 重构项目在开发执行入口层面的**唯一正式引用入口**。

以后凡是：
- 要派子 Agent
- 要启动开发
- 要调整业务
- 要改接口
- 要动数据库
- 要做联调测试
- 要做验收

都必须先回到本清单，再按本清单指向的正式基线执行。

**任何旧稿、聊天、截图、代码现状，都不能越过本清单直接成为正式依据。**
